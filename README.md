# RetailRocket E-commerce Intelligence Platform

Implementation spec for Claude Code. Read this fully before writing any code. Follow the phases in order, don't add scope beyond what's listed here.

## 0. Coding style — read before writing anything

This is a personal portfolio project for a 3rd-year student applying to Data Engineer internships, not an enterprise product. Code should look like it was written by a self-taught individual, not bulk-generated by AI. Specifically:

- **Don't add docstrings/comments to every function.** Only comment where the logic isn't self-explanatory (point-in-time join, session timeout window, feature leakage avoidance). Simple functions (getters, thin wrappers) don't need docstrings.
- **Don't wrap everything in try/except.** Only handle errors at real input/output boundaries (file reads, API calls, DB connections). Internal transform logic doesn't need defensive code for its own sake.
- **Don't use AI-sounding names** ("EnhancedProcessor", "RobustValidator", "ComprehensivePipeline", "UtilityHelperManager"). Keep names short and practical: `session_builder.py`, `funnel_agg.py`, `train_recommender.py`.
- **Don't write marketing-style docstrings/READMEs** ("This cutting-edge solution leverages..."). Keep it short and factual — state what it does, not how impressive it is.
- **A few hardcoded values with a one-line comment are fine** (e.g. session timeout = 1800s as a constant with a note on why), no need to config-ify everything like a production system.
- **Don't over-abstract.** No factory patterns, no dependency injection for an intern-scale project. Keep classes/functions flat and traceable.
- **Git history should look natural**: commit in small phase-sized chunks (setup → bronze → silver → gold → dbt → ml → api → dashboard → monitoring), short messages like `add session reconstruction logic`, `fix point-in-time join bug` — not one giant "Implement complete platform" commit.
- **A few real TODOs/notes in code are normal** (`# TODO: handle case where item_properties is missing category`) — code doesn't need to be 100% polished.
- Code comments and identifiers in English. Documentation (see section 8) also in English.

## 1. Project goal

A platform analyzing e-commerce user behavior (view → cart → purchase) on the RetailRocket dataset, covering:
1. Funnel/conversion analysis by category and time
2. Product recommendation (item-based)
3. Cart abandonment prediction
4. A dashboard with pipeline health monitoring

Business framing to use later in the case study / CV:
- Reduced cart abandonment by targeting high-risk visitors
- Improved conversion via recommendations based on co-purchase patterns
- Handled point-in-time joins on ~20M rows of time-varying item properties to avoid data leakage in feature engineering

## 2. Dataset

RetailRocket E-commerce Dataset (Kaggle: retailrocket/ecommerce-dataset)
- `events.csv` (~2.7M rows): `timestamp, visitorid, event, itemid, transactionid`
- `item_properties_part1.csv` + `part2.csv` (~20M rows total): `timestamp, itemid, property, value` — property and value are hashed, snapshots change over time
- `category_tree.csv`: `categoryid, parentid`

Important: `item_properties` is slowly-changing — each item has multiple snapshots over time. When joining with events, use the property snapshot valid **at or before** the event's timestamp (point-in-time join), never the latest snapshot — that would leak future information into training features.

## 3. Tech stack

| Layer | Technology |
|---|---|
| Orchestration | Airflow |
| Processing | PySpark |
| Recommendation | ALS (Spark MLlib) as baseline, structured so item2vec / co-occurrence-based approaches can be swapped in as alternatives |
| Abandonment prediction | XGBoost as baseline classifier, structured so LightGBM / RandomForest / logistic regression can be swapped in via config, not hardcoded as the only option |
| Bronze/Silver storage | Parquet (local filesystem, partitioned by date) |
| Gold storage | Postgres |
| Gold layer modeling | dbt-postgres |
| Serving | FastAPI + Pydantic |
| Dashboard | React + Vite + Recharts |
| Monitoring | Prometheus (scrapes FastAPI `/metrics` via `prometheus-fastapi-instrumentator`, and Airflow via statsd exporter) + Grafana for dashboards; structured JSON logging; `pipeline_runs` table in Postgres for run-level history feeding the dashboard's health tab |
| Infra | Docker Compose (single host, covers both local dev and deployment — no cloud-specific setup) |

See `architecture.mermaid` for the full data flow across layers.

## 5. Implementation phases

### Phase 1 — Bronze layer
- PySpark job reads `events.csv`, `item_properties_part1/2.csv`, `category_tree.csv`
- Write to Parquet, partition `events` by date (derived from the epoch-ms timestamp)
- Log rows read/written and duration to `pipeline_runs`

### Phase 2 — Silver layer
- Dedupe events (the raw dataset has real duplicates)
- Point-in-time join: for each event, take the property snapshot valid **at or before** that event's timestamp (Spark window function, `lag`/asof-style logic — NOT a plain join followed by taking max, which leaks future values)
- Comment this join clearly — it's the most important piece of engineering in the pipeline

### Phase 3 — Session reconstruction
- Group by `visitorid`, sort by timestamp
- New session when the gap between consecutive events exceeds 1800s (30 min) — window function, not a Python loop
- Output: `session_id, visitorid, start_time, end_time, event_count, has_purchase`

### Phase 4 — Gold layer (dbt + Postgres)
- `fct_sessions`: from session reconstruction
- `fct_funnel`: view/cart/purchase aggregated by category + day
- `dim_items`, `dim_categories`: from category_tree + latest item snapshot
- `feature_table`: item popularity, category conversion rate, co-view/co-purchase pairs (feeds both the recommender and the abandonment model)
- dbt tests: `not_null`, `unique` on primary keys, `relationships` between fct and dim tables

### Phase 5 — Modeling (keep this flexible, not locked to one algorithm)
- **Recommendation**: implement ALS (Spark MLlib) as the baseline using implicit feedback (visitorid x itemid, weighted by event type: view=1, addtocart=3, transaction=5 — comment why these weights). Structure the training script so an alternative approach (item2vec, plain co-occurrence) can be dropped in behind the same interface — implement at least one alternative if time allows, otherwise leave a clear extension point.
- **Abandonment prediction**: label = session had an addtocart event but no transaction. Split train/test by time, not randomly, to avoid leakage. Train XGBoost as the baseline, but read the algorithm choice from `ml/abandonment/config.yaml` so swapping to LightGBM/RandomForest/logistic regression doesn't require touching the training loop — evaluate at least one alternative and note the comparison in `docs/ml.md`.

### Phase 6 — API
- `GET /recommend/{item_id}` — top-N related items (from the recommender, or co-purchase fallback for cold-start items)
- `GET /funnel-stats?category_id=&from=&to=` — funnel conversion by filter
- `POST /predict-abandon` — takes session features, returns abandonment probability
- `GET /pipeline-health` — reads `pipeline_runs`, returns latest run per task
- `/metrics` — exposed via `prometheus-fastapi-instrumentator` for Prometheus scraping
- Middleware logs each request (method, path, status, latency) as JSON

### Phase 7 — Dashboard
- Funnel chart by category (Recharts bar/funnel)
- Top items by conversion rate
- Recommend demo: input an item_id, show recommended items
- Pipeline health tab: table of latest runs, status, rows processed, duration
- Component structure follows the same pattern as E-Metric Hub (Vite, components split by function, fetch wrappers isolated under `src/api/`)

### Phase 8 — Monitoring/Logging
- `logging_conf.py`: shared JSON formatter for FastAPI; minimal Python logging in PySpark jobs (no need to over-engineer this part)
- `pipeline_runs` table: `run_id, task_name, rows_processed, duration_seconds, status, started_at, error_message`
- Prometheus: scrape FastAPI `/metrics` (request latency, error rate, request count) and Airflow via statsd exporter (DAG run status, task duration)
- Grafana: at least one dashboard combining API metrics and DAG run health
- Airflow task failure triggers a simple webhook alert (POST to a URL from `.env`, e.g. a personal Discord/Slack webhook)

### Phase 9 — Infra
- `docker-compose.yml`: Postgres, Airflow (webserver + scheduler), FastAPI, Prometheus, Grafana on a shared network
- Volume mounts for Parquet data and the dbt project
- `.env.example` listing required variables (DB credentials, webhook URL)
- `docs/setup.md` covers how to run everything locally (`docker compose up`, trigger the DAG, seed data) — no cloud-specific deployment steps needed

## 6. Documentation requirements

Beyond code, produce the following under `docs/` — these matter as much as the code itself, since they're what gets shown alongside the project:

- **`docs/architecture.md`** — overview of the system, references `architecture.mermaid`, explains why each tech choice was made (short, factual, not promotional)
- **`docs/data-flow.md`** — step-by-step narrative of how data moves through the system: what Bronze looks like, what changes in Silver and why (point-in-time join explained in plain language), how sessions are built, what each Gold table represents, and how features flow into the two models. This should be detailed enough that someone unfamiliar with the code can follow the reasoning at each step, not just the schema.
- **`docs/api.md`** — endpoint reference: method, path, params, request/response examples for each route in `api/routers/`
- **`docs/ml.md`** — for each model: features used, training/test split strategy, evaluation metrics, and a short comparison if an alternative algorithm was tried (per Phase 5)
- **`docs/monitoring.md`** — what's tracked, where (Prometheus metric names, Grafana panel list, what's in `pipeline_runs`), and how to read the Grafana dashboard during a demo
- **`docs/setup.md`** — how to run the full stack locally end to end, including seeding sample data and triggering the first DAG run

Keep each doc concise and structured with headers — not a wall of text, not restating the code line by line.

## 7. Extension points (leave room, don't build now)

- JSON log format from day one → easy to attach Filebeat/Fluentd + a log aggregator later
- `pipeline_runs` table → can later be exported as additional Prometheus metrics without a schema change
- Recommender interface designed to accept a second algorithm → session-based models (e.g. GRU4Rec-style) could be added later without touching the API contract
- Feature table is currently daily batch → schema should allow adding a streaming feature source later without a full redesign (keep it simple, just don't hardcode assumptions that block this)

## 8. Out of scope for this version

- No Kubernetes, no real multi-node Spark cluster (local/standalone mode is enough)
- No complex auth for the API (basic JWT is a nice-to-have, not required)
- No real-time streaming (Kafka) — daily batch is sufficient for this scope
- No cloud-specific deployment scripting — Docker Compose on a single host is the target
